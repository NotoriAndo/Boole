# ZK.0 offline feasibility spike

Throwaway research harness for **L1 master §ZK / ADR-0017**. It answers one
question before any consensus code is written: *is a hash-generated ZK R1CS
underconstraint-detection family a sound base mining primitive, or is it an
obfuscated proof-of-work?*

**Not** wired into `self-test`, consensus, or any production path. No paid API,
no network at run time, no model calls (dev constitution §7). Only dependency is
`z3-solver` (`pip3 install --user z3-solver`).

## Run

```bash
./run.sh
```

Reproduces: determinism self-check, the S1/S2/S3/S5 gate experiment
(`experiment.py` → `result.sample.json`), and the salvage probe
(`salvage_probe.py`).

## Files

| file | role |
|---|---|
| `zkfield.py` | prime-field arithmetic, Mersenne moduli |
| `circuit.py` | R1CS model, gadget catalog, deterministic seed→circuit generator, native verifier, underconstraint-certificate check |
| `attacker.py` | two attackers: Z3 SMT search, and the O(n) solver-free **propagation attack** |
| `salvage_probe.py` | the ADR-mandated "one redesign retry": checkpoint-inversion structure, and why it also fails |
| `experiment.py` | S1/S2/S3/S5 gate metrics → JSON |
| `result.sample.json` | a committed sample run (regenerated by `run.sh`) |

## Gates (see `../../../local-docs/zk-family-phase0-report.md` for the writeup)

- **S1** shortcut resistance — strongest attacker < 1s across all bands ⇒ no-go
- **S2** verification asymmetry — attack/verify ≥ 100×
- **S3** difficulty-axis monotonicity — n/d/k control solve time
- **S5** cherry-picking — best-of-N seed grinding vs target difficulty

## Headline result

**NO-GO.** A feed-forward gadget circuit is broken in O(n) by forward
re-evaluation (no solver), independent of the difficulty band; the salvage
structure that makes Z3 time out is inverted in ~1µs by a structure-aware
attacker. Root cause: a deterministic open-source generator cannot hide
"planted freedom" from an attacker who knows the generator.
